User Controls - Part 1: What are They?

September 8th, 2007     |     

When you think of controls, your mind probably draws up images of buttons, checkboxes, listboxes, and other standard UI elements. These elements enjoy an extraordinary level of support in WPF. You can modify how these elements look using styles, templates, and skins, and you can use data binding to bind them to some data. These built-in controls make up much of what most people will use in their applications.

There will be numerous scenarios, though, where the built-in controls are simply not enough. Or, as it may be, the built-in controls are a bit too much with extra unnecessary functionality. In such cases, what you can do is create your own controls known as User Controls. In this post, I will give you a brief introduction to user controls and how similar they are to normal, built-in controls.

Sample User Control - InfoRectangle
Before I go on, let me first provide you with an example of a user control that I created. The following image shows you my user control called InfoRectangle:

The InfoRectangle user control is pretty simple. It is a rectangle with curved edges, and inside that rectangle, there is a label where text is displayed. Just like any other control, I can use and modify it within my application as shown below:

Now that you have a brief, though fuzzy, idea of what user controls are, let's take a better look at what they are.

What are User Controls?
You can think of user controls as self-contained mini-applications (kind of like a widget) composed of a XAML and code-behind file. For example, our InfoRectangle user control is made up of both the InfoRectangle.xaml file where the interface is declared and the InfoRectangle.xaml.cs file where some procedural code defines some of our control's properties:

One way of thinking about user controls is to compare them to inline frames in a HTML page. A parent HTML page contains information on where to place an inline frame. Inside that inline frame, you display another web page. The web page you display inside the frame can be quite simple, or the displayed page can be quite complex with its own navigation links, scripts, and more. What is guaranteed, though, is that the parent HTML page hosting the inline frame does not care much about the page displayed inside the inline frame.

Let's look at the following page I created:

The page I created is quite simple. What is displayed inside my inline frame, though, is not so simple, but I have no control as to how the external site works.

To bring the discussion back to WPF, your parent HTML page is similar to a parent control such as a layout control. The inline frame is similar to your user control. Your parent control's code is separate from that of the code that makes up your user control. In fact, your user control is defined in separate files, and it may even have been created by somebody else! All your parent control does is take care of the paperwork needed to display the user control and maybe push a few buttons as needed. It never opens the hood of your user control to fiddle with the internals.

Using User Controls
Despite how simple or complex your user controls may be, using them in your applications is straightforward. There are various ways of using user controls, and while we all have our preferences, there is no one good way to incorporate user controls in your applications. Instead, let's take a look at the three most common ways.

The following is how you could use the InfoRectangle control using C#:

InfoRectangle ifr = new InfoRectangle();
ifr.QuickInfo = "All in C#";
 
LayoutRoot.Children.Add(ifr);

In the above code, QuickInfo is the name of a public property that my control exposes, and the value you pass to it determines what text gets displayed in your InfoRectangle object. If you don't feel like using procedural code, you can easily use this control using XAML:

<CustomControl:InfoRectangle QuickInfo="All About XAML"/>

The XAML declaration is a bit more clean, but notice that nothing extra is being done. The end result is just the same.

Finally, if you shun coding altogether, you can add this control using Expression Blend's WYSIWYG tools by selecting your user control from the Asset Library and adding it to the Artboard:

When on your Artboard, you can manipulate your InfoRectangle just like any other design element or control:

In the above image, I added one InfoRectangle object, created two copies of that object, and finally changed their QuickInfo text and background color. All of this was done without writing a single line of XAML or C# code.

This Land is Your Land, This Land is My Land
The main takeaway message from this post is that you shouldn't discriminate between user controls and built-in controls. In the grand scheme of things when you step back and forget about the behind-the-scenes implementation, your user control is capable of being used just like a regular built-in control. To nitpick, being "capable" does not necessarily mean that is how your user control will be. You need to make a conscious effort to ensure your user control takes advantage of features that built-in controls provide.

Hopefully this post gave you an introduction to what user controls are. Subsequent posts will discuss some interesting features such as dependency properties, routed events, etc. that could play a role in how you design your user controls.

Cheers!
