SQL + Images

I have a question for all the SQL experts... In the Impilo Health Management project, I want to create a browser of all the existing gym equipment. I created tables in my database to store information about these machines (ie. Description, instructions, manufacture, etc). I also need a image or thumbnail of this machine to be displayed using a LINQ query? Should I:

  1. Store the image in the SQL database and bind it to the Image object
  2. Store a path name to a image and bind this to the Image URI?

For more information visit www.codeplex.com/impilo

# re: SQL + Images

Being a data-centric person I'd recommend putting it in the database as data integrity can be maintained as well as what happens when you put in a new application server to balance the load, where does it get the image from? Do you have a shared drive between all application servers creating a performance bottleneck or do you write it to a drive on each application server?

I prefer the approach of popping it into the database and dealing with the scalability issue in a uniform manner.

# re: SQL + Images

You have 2 options:

A. Put it in the database which is easy for back-ups and such. Do put them on a seperate filegroup if your using sql server and make sure that you'r not using a clustered index.

B. Put them on a local directory and if replication is important then make use of the Windows server build in replication possibilities. If your using the right naming convention then you don't even need to have these files located in the database.

Only problem you have with solution B is what to do with these files if they should be part of a transaction? One way to fix this is to have a index in a database and do a scheduled clean-up where you remove files from the filesystem that arent registered in the index table. Ofcourse, forget about update at file level. Only delete/insert kinda stuff is allowed in such scenarios.

# re: SQL + Images

Go with A.  Another point that has not been raised is that of backups:  With the data in the database, it will get backed up with the database.  Much easier to restore if there's a failure.

Option B is very unstable.  Say you store the files remotely, and the DB fails; which version of files/db is correctly matched?

Basically A should be the only option.

# re: SQL + Images

I've worked on a couple of Imaging products, and there's no doubt in my mind that for any large set of images (more than a couple thousand), the images should be stored on local disk.

One of the most important reasons is simply for disk and backup scheduling. When the image store gets huge, it's a pain to back up, so we generally put images on RAID 5 sets for reliability, and then do full back ups less often. The database then goes on RAID0+1 sets and is backed up frequently.

Sure, one can create separate filegroups on SQL Server, and place the relevant columns into the relevant filegroups, but very few servers have two separate RAID clusters.
